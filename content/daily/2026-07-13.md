## 今日 AI 重點

今天最值得學的不是某一個模型，而是 agent skills 正從零散提示詞走向跨工具的可攜工作單位。GitHub Trending 可見 Stitch Skills、Superpowers、Claude Code templates、DesktopCommanderMCP 與 Claude cookbooks：開發者正在把工作流程、領域知識與工具操作封裝成可重用 skills。

這也形成新的供應鏈邊界。NVIDIA SkillSpector 所引用的大規模研究掃描 42,447 個 marketplace skills，發現 26.1% 至少含一種弱點、5.2% 出現高度疑似惡意模式；含可執行腳本的 skills 更容易出問題。真正的實務能力，是建立 Skill Intake Gate：確認來源與內容、列出權限、掃描、最小權限 sandbox 試跑、核准、留 run log，最後才進正式 workflow。

## 10 分鐘深讀

### 發生什麼

Agent skills 正在成為新的軟體分發層。它們把提示、流程、工具設定、腳本與依賴包成可重用單位，使同一套工作方法能跨 Codex、Claude Code、Gemini CLI、Copilot 或其他 agent harness 複用。Trending 的 skills、workflow framework 與 MCP terminal 工具顯示，開發者正把 agent 從單次聊天拉向可組裝的工作系統。

### 為什麼現在重要

可攜性會放大價值，也會放大風險。一份有問題的 skill 若能被多種 agent 直接安裝，可能攜帶 prompt injection、資料外洩、權限提升、惡意 script 或供應鏈依賴。SkillSpector 所引用的研究讓這不是理論問題：風險模式已經能在 public marketplace 被觀察到。

### 技術背景

Skill Intake Gate 至少有六步：provenance（publisher、release / commit、授權與維護狀態）、content review（SKILL.md、腳本、依賴與外部 URL）、capability inventory（file、shell、network、credential、browser、external send）、static scan、sandbox trial，以及 approval and observability（owner、版本 pin、run log、revocation）。

### 企業影響

金融業泛化場景中，skills 可快速複用到文件整理、知識查詢、客服草稿、報表生成或開發自動化。正確做法不是禁止，而是分級導入：instruction-only 與 read-only retrieval 可較快試驗；含 script、network、write action、customer-facing 或敏感資料存取的 skills 必須進 allowlist、sandbox、approval 與 log。速度與稽核不必二選一。

### 後續觀察

觀察 SKILL.md / Agent Skills standard 是否加入 permission manifest；Codex、Claude Code、Copilot、Gemini CLI 是否提供 provenance 與掃描；MCP server 與 skills 的權限模型能否一致；以及市場是否出現 SBOM、signature、risk score 與 revocation feed。

## 大家正在用 AI 做什麼

## Codex Pro 額度最大化

使用原則：把額度投入可長期複用的 Skill Intake Gate：manifest、掃描整合、sandbox 測試、allowlist 與文件；不要把額度花在未驗證 skills 的反覆試跑。

最值得投入的專案：

- Skill Intake Gate Dashboard：顯示 publisher、permission、scan finding、approval、owner 與 expiry。
- Agent Capability Manifest Kit：以 JSON 定義每個 skill 的檔案、網路、shell、credential 與 external action 邊界。
- Sandboxed Skill Test Harness：以 fixture repo 和假資料驗證 skill 的實際檔案、命令與網路行為。

本週配置：

- 第 1 段：定義 capability manifest 與四級 trust model。
- 第 2 段：做 mock skill 掃描結果頁，加入 allow / review / reject 決策。
- 第 3 段：補 README、threat model、測試案例與 AI FDE 面試說明。

避免浪費：

- 不要因為 repo 熱度高就直接給 skill shell、network 或 credential 權限。
- 不要把 scanner 無 finding 解讀為 production-safe。
- 不要讓不明 skill 直接操作真實 repo、信箱、客戶資料或外部系統。

來源：https://github.com/nvidia/skillspector、https://arxiv.org/abs/2601.10338

## 重要名詞

**Skill Intake Gate**

白話解釋：

Skill Intake Gate 是在安裝或啟用 agent skill 前，依序確認來源、內容、權限、掃描、sandbox、核准與可觀測性的控制流程。

實務例子：

報表摘要 skill 可以先在假資料與 read-only 權限下試跑；若它要執行 shell、下載依賴或寄出郵件，就必須額外申請 network、external-send 與人為批准。

常見誤解：

它不是單一安全掃描工具；掃描只是其中一關，仍需要最小權限、版本 pin、run log 與撤銷機制。

## AI FDE 轉職練習

客戶情境：

客戶想讓多個團隊直接從 public marketplace 安裝 agent skills，加速文件整理、資料查詢、程式協作與工作流自動化；資安與法遵主管則擔心不明腳本、資料外洩與權限失控。

需求釐清：

- skill 的 publisher、版本、依賴、腳本與實際需要的 file / network / credential 權限是什麼？
- 哪些工作可用 instruction-only 或 read-only skill 先做 sandbox PoC，哪些必須進 allowlist 與人工批准？
- 若掃描出新 finding 或 upstream 被接管，誰負責 revoke、通知使用者與回退 workflow？

今日小練習：

為三種 skill 設計 Trust Level：L1 instruction-only、L2 read-only tool、L3 scripted / network、L4 external-write / sensitive-data；為每級寫出必備證據與核准者。

面試可用表述：

我會把 agent skill 視為新的軟體供應鏈，而不是單純提示詞。我的導入方式是先建立 Skill Intake Gate：provenance、權限清單、靜態掃描、sandbox、owner、approval 與 revocation，讓團隊能安全地加速複用。
