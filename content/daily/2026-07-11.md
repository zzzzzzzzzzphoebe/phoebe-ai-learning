## 今日 AI 重點

今天最值得學的不是哪個 agent 又能多做一個 demo，而是 agent 的交付標準正在改變：從「幫我做完」升級成「留下可驗證證據」。Paper-replication workflow 把 coding agent 複現論文的每個 claim 拆成 target、provenance、comparison 與 validation gate；Prismata 把 web agent 面對第三方內容時的 prompt injection 風險拆成 contextual least privilege；HalluSquatting 則提醒我們，agent 不能把幻覺出來的 repo、package 或 URL 當成可執行事實。

對 AI FDE 來說，這是一個很實用的轉向。企業不會只問 agent 會不會完成任務，而會問來源是否可信、權限是否最小化、成本是否可控、結果是否可重跑、錯了能不能回退。今天要把這些訊號轉成一張 Agent Evidence Trail Checklist，讓你能在作品集與面試裡說明「我不只會用 agent，我會把 agent 做成可驗收的工作系統」。

## 10 分鐘深讀

### 發生什麼

今天的共同訊號是：agent 已經不只是在聊天視窗裡給建議，而是開始讀 repo、跑指令、複現研究、操作網頁、提交表單、接 CRM，甚至可能抓取外部 package。當 agent 的能力從文字輸出變成行動能力，交付標準也必須跟著升級。真正有價值的不是 agent 說「我完成了」，而是它留下什麼證據能讓人驗收。

### 為什麼現在重要

Paper-replication 顯示研究複現可以被拆成 claim targets 與 validation gates；Prismata 顯示 web agent 必須面對 untrusted page content；HalluSquatting 顯示模型幻覺可以變成供應鏈攻擊；HN 的 MCP quote-action 則顯示小型企業已經開始讓 agent 觸發 CRM 動作。這些訊號合在一起，代表企業導入 AI 的問題不再是「agent 能不能做」，而是「做的過程是否可信」。

### 技術背景

一個 evidence-first agent workflow 至少有八層：

- task target：把需求拆成可驗收 claims 或 acceptance criteria。
- source verification：確認引用的 repo、package、文件、模型與資料集是真實且可信的。
- tool boundary：限制 shell、browser、network、email、CRM、filesystem 等工具能力。
- provenance：記錄每個輸入、來源、命令、artifact 與產出路徑。
- validation：使用 tests、linters、checks、comparison tables 或人工抽查驗證。
- cost routing：讓高風險任務使用強模型，低風險高頻任務使用穩定便宜模型。
- human approval：對寫入、寄送、上傳、刪除、付款與客戶互動做 gating。
- rollback：讓錯誤輸出或錯誤動作能停止、回復與補救。

### 企業影響

金融業泛化場景尤其需要這套思維。若 agent 用於授信文件整理、客服草稿、內部知識查詢、法遵初篩或開發流程，企業不會只問它能不能產生答案，而會問答案根據哪裡、是否引用正確資料、是否觸碰敏感欄位、是否有人工覆核、錯誤時能否追溯。AI FDE 的價值，就是把這些問題轉成可執行的 workflow、dashboard、checklist 與 demo。

### 後續觀察

接下來觀察四件事：第一，coding agent 是否開始內建 source verification 與 dependency trust gate；第二，web agent 防護是否從 prompt 層走向 browser/runtime 層；第三，MCP / agent action 是否會出現更清楚的 read/write permission pattern；第四，模型供應商是否提供更穩定的便宜模型層與 migration runway，讓企業可以做長期成本規劃。

## 熱門討論

- 支持觀點：如果 agent 能把任務、來源、工具呼叫、產出與檢查全部留下來，它會比傳統手工作業更可追溯，也更容易持續改善。
- 疑慮風險：如果 agent 可以自行瀏覽、安裝、提交表單或寫入系統，幻覺、prompt injection 與權限過大會把小錯誤放大成流程風險。
- 後續追蹤：觀察 coding agent、browser agent 與 MCP tools 是否開始內建 source verification、least privilege、action approval 與 cost routing。

## 重要名詞

**Evidence Trail**

白話解釋：Evidence trail 是 agent 完成任務時留下的可驗證紀錄，包含需求拆解、來源、工具呼叫、產出檔案、檢查結果、人工批准與回退方式。

實務例子：Codex 更新網站時，不只回報完成，還要有資料來源、修改檔案、node --check 結果、git diff、commit hash、push 狀態與 Gmail send result。

常見誤解：Evidence trail 不是把 log 全部丟給人看，而是留下能回答「為什麼可信、如何重跑、錯了怎麼補救」的最小必要證據。

## AI FDE 轉職練習

客戶情境：客戶想讓 AI agent 自動讀取公開資料、更新內部知識庫，並在符合條件時把 lead 或案件送進 CRM，但資安與法遵擔心 agent 被網頁內容誘導或引用錯誤來源。

需求釐清：

- 哪些工具是 read-only，哪些工具會寫入 CRM、寄信、建立案件或觸發外部動作？
- agent 引用的 repo、package、文件與網站來源要如何驗證，是否有 allowlist 或人工批准？
- 每次 agent action 是否能保留 task target、source、tool call、artifact、validation 與 rollback 紀錄？

今日小練習：用 30 分鐘寫一張 Agent Evidence Trail Checklist，並拿一個 public-safe 流程測試：例如更新學習站、建立 FAQ、或提交模擬 lead。

面試可用表述：我不會只把 agent 接進業務系統，而會先把 read-only knowledge、write action、source verification、least privilege、人工覆核與 evidence trail 設計出來；這樣 agent 做得越多，系統反而越可追溯。
